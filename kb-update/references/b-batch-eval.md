# B 模块：批量融合评估

## 执行流程

```
Step 1: 读取待评估池（双轨验证）
  → 收集所有需评估的来源摘要

Step 2: 按适用模块分组
  → 读取每篇来源摘要的 `适用SOP` + `适用概念`
  → 按模块分组，形成"模块 → 待评估资料列表"

Step 3: 逐组评估（核心）
  → 读取模块当前内容
  → 读取该组所有关联新资料的内容
  → 按 5 维度判断：能否让模块"变得更好"？

Step 4: 产出融合建议报告
  → 每个模块一份评估结果

Step 5: 用户确认 + 融入执行
  → 用户逐条确认：融入 / 忽略 / 待观察
  → 执行 Write/Edit
  → 更新状态
  → 调用 kb-check
```

---

## Step 1：读取待评估池

**双轨扫描**（两条轨道的结果取并集）：

**轨道 1（动态扫描）**：
```bash
# 逻辑伪代码
for each 来源摘要 in 知识库/来源摘要/:
    source_date = 来源摘要.frontmatter.最后更新
    for each module in (来源摘要.frontmatter.适用SOP + 来源摘要.frontmatter.适用概念):
        module_path = 知识库/{module}.md
        module_date = module.frontmatter.最后更新
        if source_date > module_date:
            if 来源摘要.frontmatter.评估状态 not in ["已评估-无价值", "已评估-已融入"]:
                加入待评估池
```

**轨道 2（状态字段）**：
- 扫描所有 `评估状态: 未评估` 的来源摘要
- 加入待评估池

**冲突处理**：
- 轨道 1 认为需评估，轨道 2 标记 `已评估-无价值` → 跳过（人工已判定）
- 轨道 1 认为需评估，轨道 2 标记 `已评估-已融入` → 跳过
- 轨道 1 认为需评估，轨道 2 标记 `已评估-待观察` → 保留在池中（继续提醒）
- 轨道 1 认为不需评估，轨道 2 标记 `未评估` → 保留在池中（安全优先）

---

## Step 2：按适用模块分组

读取每篇待评估来源摘要的 frontmatter：
- `适用SOP` → 按 SOP 分组
- `适用概念` → 按概念页分组

形成分组表：

```markdown
| 模块 | 关联来源摘要 | 来源可信度 |
|------|------------|-----------|
| SOP-4 | 喃姐日志-9个正确做GEO、硅基动力-可引用密度、姚金刚-GEO内容 | L3, L3, L2 |
| 概念/白帽GEO | 喃姐日志-9个正确做GEO | L3 |
```

---

## Step 3：逐组评估（核心）

**评估前准备**：
1. Read 模块当前完整内容
2. Read 该组所有关联来源摘要的完整内容

**评估标准（5 维度）**：

| 维度 | 判断标准 | 权重 |
|------|---------|------|
| **完整性** | 模块是否缺少某个子主题？新资料补充了？ | 高 |
| **精确性** | 现有阈值是否模糊（"适当""合理"）？新资料提供了量化数据？ | 高 |
| **可操作性** | 现有指南是否太抽象？新资料给了具体步骤/公式/模板？ | 中 |
| **证据强度** | 现有论断基于 L3，新资料提供 L2/L1 支撑？ | 中 |
| **时效性** | 现有数据/案例是否过时？新资料提供了更新的？ | 低 |

**评估输出格式**（每条优化点）：

```markdown
### #{编号} {融入类型}：{简述}

**维度**：{完整性/精确性/可操作性/证据强度/时效性}
**位置**：{模块} §{节号}

**现有内容**：
{原文引用，50 字以内}

**新资料补充**：
{来源摘要名称} — {核心补充内容}

**融入效果预览**（diff）：
```diff
--- 当前内容
+++ 融入后
@@
{具体 diff}
```

**评估理由**：
{为什么这是"变得更好"，而非"不一样"}

**影响分析**：
{对上下游模块的影响，如 SOP-5 质检需同步调整}

**建议动作**：{建议融入 / 忽略 / 待观察}
```

**评估结果标记**：
- `建议融入` — 有明确优化价值，diff 预览已生成
- `忽略` — 无优化价值，或已有更高可信度来源覆盖
- `待观察` — 有价值但需更多佐证（如仅 1 个 L3 来源，需等待 L2 来源交叉验证）

## Step 4：产出融合建议报告

**报告结构**：

```markdown
---
type: AI操作记录
audit_type: kb-update
date: YYYY-MM-DD
module_count: {评估模块数}
source_count: {评估来源数}
---

# kb-update 融合建议报告 — {日期}

## 评估概览

| 指标 | 数值 |
|------|------|
| 评估模块 | N 个 |
| 关联来源 | M 篇 |
| 建议融入 | X 项 |
| 建议忽略 | Y 项 |
| 待观察 | Z 项 |

## 按模块分组

### {模块名}

**关联来源**：
- [[来源摘要/xxx]]（L{层级}）

**当前状态**：{模块状态摘要}

#### #{编号} [{融入类型}] {简述}
...
（逐条优化点）

## 优先级排序

| 优先级 | 编号 | 模块 | 维度 | 简述 |
|--------|------|------|------|------|
| P1 | #1 | SOP-4 | 完整性 | 新增 §2.5 合规标准 |
| P1 | #2 | SOP-1 | 精确性 | 补充 AI 可见性量化阈值 |
| P2 | #3 | 概念/可引用密度 | 可操作性 | 补充四类句型示例 |

## 推荐执行顺序

1. **P1 项**（建议立即融入）
2. **P2 项**（建议本周内融入）
3. **待定项**（待观察或需更多佐证）
```

---

## Step 5：用户确认 + 融入执行

**融入确认交互**：

```
AI：报告已写入 AI操作记录/kb-update-2026-04-27.md。
    建议融入 3 项，请确认：

    #1 [新增] SOP-4 §2.4 后新增 §2.5 合规标准
       → diff 预览见报告 §2.1
    #2 [修改] SOP-4 §2.3 补充量化阈值
       → diff 预览见报告 §2.2
    #3 [补充] SOP-1 §2.2 追加新来源标注
       → diff 预览见报告 §3.1

    忽略：2 项 | 待观察：1 项

用户：#1 #2 融入，#3 忽略，待观察的保持
AI：执行 #1 新增 §2.5... ✅ 完成
    执行 #2 修改 §2.3... ✅ 完成
    更新来源摘要「喃姐日志」评估状态 → 已评估-已融入
    更新来源摘要「硅基动力」评估状态 → 已评估-已融入
    更新 SOP-4 最后更新 → 2026-04-27
    运行 kb-check 全量审计...
```

**融入后状态更新**：
- 已融入的来源摘要：`评估状态: 已评估-已融入`
- 忽略的来源摘要：`评估状态: 已评估-无价值`
- 待观察的来源摘要：`评估状态: 已评估-待观察`
- 融入的模块：`最后更新: {当前日期}`

**融入后强制质检**：
- 调用 `kb-check` 全量检查
- 不通过 = 回滚融入内容，标记为 `已评估-待修复`
