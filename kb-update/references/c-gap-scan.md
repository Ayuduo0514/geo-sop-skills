# C 模块：缺口反向扫描

## 定位

不从新资料出发，从**存量知识的弱点**出发。

**与 kb-check 的边界**：

| | kb-check | kb-update C |
|--|----------|-------------|
| 性质 | 错误/缺陷 | 机会/优化 |
| 例子 | SOP 缺 §3（结构缺陷） | SOP §2 有"适当频率"但无量化 |
| 强制度 | 必须修复 | 建议改进，可做可不做 |

---

## 执行流程

```
Step 1: 扫描所有 SOP/概念页的 5 类弱点
Step 2: 反向匹配来源摘要池
Step 3: 产出缺口-填补匹配表
Step 4: 用户确认后，把候选来源纳入 B 模块待评估池
```

---

## Step 1：扫描 5 类弱点

**扫描目标**：`知识库/综合/*.md`（SOP）+ `知识库/概念/*.md`

**弱点类型与识别方法**：

| 类型 | 识别方法 | 扫描工具 |
|------|---------|---------|
| **模糊阈值** | 文本中出现"适当""合理""较高""建议""通常" + 无量化数据支撑 | Grep 关键词列表 |
| **无来源论断** | 数字/%/具体数据后无 `来源[Lx]` 或 `[[来源摘要/` | 正则：`(\\d+%?|\\d+\\.\\d+)` 后无 `来源[` |
| **定义不完整** | 概念页缺少关键维度（如只定义了"是什么"没定义"怎么用""边界在哪"） | Read 后判断 |
| **引用层级低** | 核心论断仅靠 L3 来源支撑 | Read 后判断 |
| **结构空洞** | 有标题但内容明显过短（<50 字，占位符级） | 字数统计 |

**模糊阈值关键词列表**（Grep 用）：
```
适当、合理、较高、较低、建议、通常、一般、适度、适量、尽量、较好、较差
```

**无来源论断检测**：
```bash
# Bash：查找数字/百分比后无来源标注的行
grep -n -P '(\d+%?|\d+\.\d+)[^\n]{0,50}$' *.md | grep -v "来源\["
```

---

## Step 2：反向匹配来源摘要

对每类弱点，扫描来源摘要池：

```python
for each 弱点 in SOP/概念页:
    主题词 = 提取弱点所在主题的关键词
    for each 来源摘要 in 知识库/来源摘要/:
        if 来源摘要.核心主题 匹配 主题词:
            if 来源摘要.可信度 >= 当前引用层级:
                if 来源摘要.内容 能直接填补该弱点:
                    记录为"候选填补"
```

**匹配标准**：
- 主题匹配度 ≥ 70%（关键词重叠）
- 可信度不降低（L2 来源不能填补 L1 已覆盖的内容）
- 内容可直接引用（不是泛泛而谈，有具体数据/步骤/案例）

---

## Step 3：产出缺口-填补匹配表

**报告结构**：

```markdown
---
type: AI操作记录
audit_type: kb-update-gap-scan
date: YYYY-MM-DD
---

# kb-update 缺口扫描报告 — {日期}

## 扫描概览

| 指标 | 数值 |
|------|------|
| 扫描 SOP | N 个 |
| 扫描概念页 | M 个 |
| 发现弱点 | X 个 |
| 可填补候选 | Y 个 |

## 按模块分组

### {模块名}

#### 弱点 #{编号}

| 字段 | 内容 |
|------|------|
| **类型** | 模糊阈值 / 无来源论断 / 定义不完整 / 引用层级低 / 结构空洞 |
| **位置** | §{节号} / 整页 |
| **现有内容** | {原文引用} |
| **严重程度** | 高 / 中 / 低 |
| **候选填补来源** | [[来源摘要/xxx]]（L{层级}） |
| **填补内容** | {来源摘要中能填补该弱点的核心内容} |
| **优先级** | P1 / P2 / P3 |

---

## 优先级排序

| 优先级 | 模块 | 弱点类型 | 候选来源 | 简述 |
|--------|------|---------|---------|------|
| P1 | SOP-1 | 模糊阈值 | 姚金刚-GEO方法 | "AI可见性较高"→四级评分 |
| P2 | 概念/可引用密度 | 定义不完整 | 硅基动力-可引用密度 | 缺量化测量方法 |
```

---

## Step 4：与 B 模块联动

C 模块不直接融入。产出后：

1. **用户确认**：用户阅读缺口扫描报告，确认哪些弱点值得填补
2. **标记来源**：将用户确认的候选来源的 `评估状态` 重置为 `未评估`（如果当前是 `已评估-无价值`）
3. **纳入 B 模块**：用户运行 `kb-update`（B 模块）时，这些来源进入待评估池
4. **定向评估**：用户也可运行 `kb-update --focus=SOP-1`，只评估特定模块关联的来源

**或者直接执行**：如果用户确认"把 #1 #2 #3 都补上"，可以直接进入 B 模块的 Step 3-5（跳过评估，直接融入）。

---

## 扫描频率建议

| 场景 | 频率 |
|------|------|
| 知识库快速膨胀期（每周 ingest >3 篇） | 每月一次 |
| 稳定期（每月 ingest <5 篇） | 每季度一次 |
| SOP 重大重构后 | 立即一次 |

**触发词**："知识库该补补了""看看有什么缺口""反向扫描"
