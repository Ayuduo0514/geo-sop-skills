# GEO Skill — 生成式引擎优化

## 定位

为中国国内 AI 引擎（豆包、千问、Kimi、元宝、DeepSeek）优化品牌内容可见性的 multi-agent 工作流。

**输入**：品牌/产品名称（如「歌剧魅影」）  
**输出**：品牌知识库 + GEO 策略报告 + 多优先级内容包 + 质检报告 + 引用模拟报告

---

## 环境前置检查

**启动前必须先读 Schema 文件（按顺序）：**
1. `D:\Obsidian Vault\CLAUDE.md` — vault 全局约定
2. `D:\Obsidian Vault\GEO\CLAUDE.md` — GEO 域规则
3. 操作 theory-wiki 时读：`D:\Obsidian Vault\GEO\theory-wiki\CLAUDE.md`
4. 操作某品牌时读：`D:\Obsidian Vault\GEO\brands\CLAUDE.md` + `D:\Obsidian Vault\GEO\brands\{品牌名}\CLAUDE.md`（如存在）

不读 CLAUDE.md 直接操作 wiki = 违反 schema，产出无效。

```bash

ls "D:/Obsidian Vault/GEO/"
```

- Vault 根目录：`D:\Obsidian Vault\GEO\`
- 若 Obsidian 未开启，直接用 Write 工具写文件到 Vault 目录（等效）

## 内容优先级框架

| 优先级 | 类型 | 示例 | 是否生成 |
|--------|------|------|---------|
| P0 | 官网、认证账号 | 品牌官网、官方微博 | ✅ |
| P1 | 权威媒体 | 官媒、垂类头部 | ✅ |
| P2 | PGC内容 | 知乎、百家号 | ✅ |
| P3 | BGC内容 | 品牌自媒体 | ✅ |
| P4 | UGC | 用户评论、短视频 | ❌ 不处理 |

## GEO 评分标准（内置）

每篇内容按以下 4 个维度评分（各 25 分，满分 100）：

1. **直接答案密度**：AI 能否从文中直接提取问题答案，无需推断？
2. **统计数据量**：含可引用的数字、数据、来源引用数量
3. **实体清晰度**：品牌名、产品名、人物、地点等命名实体是否清晰、无歧义
4. **结构化程度**：标题层级、列表、表格使用是否便于 AI 解析

**合格线**：总分 ≥ 70，单项 ≥ 15  
**P0 格式**额外要求：实体清晰度 ≥ 22

## Agent 拓扑

```
用户输入品牌名
    ↓
Orchestrator（主控）
    ├── Researcher（并行）→ 品牌资料收集
    ├── Entity Agent（并行）→ 实体档案编译
    └── [等 Researcher + Entity 完成]
         ↓
    Strategist → GEO 策略报告
         ↓
    ┌────────────────────────┐
    │  Writer P0（并行）      │
    │  Writer P1（并行）      │
    │  Writer P2-3（并行）    │
    └────────────────────────┘
         ↓
    QC Agent → 质检（评分 + 红线检查）
         ↓
    Simulation Agent → 引用模拟测试
         ↓
    Orchestrator 汇总 → 写入 Obsidian Vault
```

### 熔断条件

Orchestrator 在以下情况**暂停并通知用户**：
1. 品牌信息严重不足（Researcher 找到的有效信息 < 30% 覆盖核心字段）→ 要资料
2. 任何内容命中品牌红线 → 人工确认后继续
3. 某 Agent 连续两次输出被下游打回 → 升级人工判断

## Orchestrator 执行流程

### Step 0：初始化

```
品牌名: {BRAND}
Vault 路径: D:\Obsidian Vault\GEO\{BRAND}\
```

检查是否有历史知识库（`wiki/{BRAND}/entity.md` 存在？）：
- **有**：加载历史知识库，进行增量更新
- **无**：全新建立

询问用户：
> 「你有 [品牌名] 的资料吗？（品牌手册、官网链接、产品文档等）有的话发给我，没有我自动搜索。」

### Step 0.5：意图词挖掘（SOP-2）

**触发判断**：
- 用户明确说"挖意图词"或"建词库" → 执行本步骤
- 用户说"做 GEO"但没有现有词库 → 执行本步骤
- 用户已有意图词库 → 跳过，直接进入 Step 1

**Phase 0：品牌信息收集**（自动执行，不需要用户提供）

用户提供品牌名后，Claude 自动搜索以下信息：
1. 品牌是 ToC 还是 ToB
2. 所属行业 / 产品品类
3. 核心产品/服务（2-3 个）
4. 主要竞品（3-5 个）
5. 品牌核心优势 / 差异化卖点

搜索方式：加载 web-access skill，搜索品牌官网、媒体报道、电商平台等。

搜索完成后，整理成简要摘要展示给用户确认：
> 「我查到 [品牌名] 的信息如下，确认一下：
> - 类型：ToC / ToB
> - 行业：XX
> - 核心产品：A、B、C
> - 竞品：X、Y、Z
> - 核心优势：...
> 信息对吗？有补充的吗？」

用户确认后进入下一步。信息不全时问用户补全。

**执行流程**（详见 `references/sop-2-intent-mining.md`）：

1. **通用五场景骨架填充**
   - 认知启蒙型、对比选购型、操作指导型、品牌评价型、风险规避型
   - 每类至少 5 个询问句
2. **L1-L5 层级标注**
   - L1 认知 → L2 探索 → L3 评估 → L4 决策 → L5 口碑
   - ToC 品牌：L3+L5 词量之和 ≥40%
3. **四要素公式补差异化词**
   - 意图词 = 需求维度 + 行业属性 + 层级标签 + 转化导向
   - 围绕品牌核心优势，生成竞品未覆盖的词
4. **人群×场景矩阵**
   - 前 3 大目标人群 × 前 5-10 个具体场景
5. **词库整理与阈值校验**
   - 每个意图标签：精准词 ≥3、长尾词 ≥5、询问句 ≥4
   - 询问句五类覆盖：是什么 / 为什么 / 怎么办 / 怎么选 / 哪家好
   - 带品牌名词占比 ≤20%
   - 冷启动词库规模：50-100 词
6. **同步到飞书多维表格**（可选）
   - 首次生成：`python scripts/sync_to_feishu.py {BRAND}-intent-library.md --mode create`
   - 增量更新：`python scripts/sync_to_feishu.py {BRAND}-intent-library.md --mode update`
   - 配置文件：`scripts/config.json`（需填入飞书 App ID、Secret、表格 token）
   - 增量逻辑：按「意图标签」upsert，旧词库有但新生成没有的标记为「待复审」

**输出**：`{BRAND}-intent-library.md`，格式见 references/sop-2-intent-mining.md 产出物模板。

**继续下一步**：意图词库完成后，进入 Step 1（研究阶段），词库为知识库提供"要回答哪些问题"的方向。

### Step 1：研究阶段（并行）

**Researcher Agent Prompt：**
```
目标：收集 [{BRAND}] 的品牌资料，用于 GEO 内容生成。

必须覆盖的字段（找不到的标注「待补充」）：
- 品牌简介（一句话定义：是什么、做什么、面向谁）
- 核心产品/服务（名称、功能、差异点）
- 品牌历史（成立时间、关键里程碑）
- 目标受众（人群画像）
- 品牌价值主张（官方口号/slogan）
- 媒体报道摘要（主流媒体如何描述它）
- 竞争格局（主要竞品，3-5 个）
- 已有的内容阵地（官网、微博、微信、抖音等）

信息来源优先级：用户提供的资料 > 官网 > 权威媒体 > 其他

必须加载 web-access skill 并遵循指引。
有联网条件时优先获取官网一手信息。
输出：结构化 JSON，每个字段附来源 URL。
```

**Entity Agent Prompt：**
```
目标：基于 Researcher 的输出，编译 [{BRAND}] 的实体档案。

核心任务：
1. 提取所有命名实体（品牌名变体、产品名、人物、地点、专有概念）
2. 统一命名（处理同一事物的多种叫法）
3. 建立关系图（实体之间的关联）
4. 标注歧义点（可能与其他实体混淆的地方）

三步编译法处理冲突资料：
- 浓缩：提取每篇资料的核心论点
- 质疑：找出资料间的矛盾，明确标注
- 对标：以官方来源为准做最终裁决

输出格式：entity.md（见下方模板）
```

### Step 2：策略报告

**Strategist Agent Prompt：**
```
基于品牌知识库，制定 [{BRAND}] 的 GEO 策略报告。

报告结构：
1. 当前 AI 引擎可见性诊断
   - 模拟 10 个用户可能问 AI 的关键 Query
   - 评估现有内容对这些 Query 的覆盖率
   
2. 内容缺口分析
   - 哪些核心 Query 目前没有高质量内容覆盖？
   
3. GEO 优化优先级
   - P0: 必须立即覆盖的 Query（影响品牌基本认知）
   - P1: 高价值机会 Query（有竞争但可切入）
   - P2: 长尾补充 Query
   
4. 各优先级内容策略
   - 每种格式（P0/P1/P2-3）分别写什么、怎么写
   - 每篇内容的目标 Query 和预期 GEO 得分

输出：strategy.md
```

### Step 3：内容生成（全并行）

三个 Writer Agent 同时运行，各自写对应格式。

**Writer Agent 通用约束：**
- 语言：中文
- 结构：多用标题、列表、表格（提升结构化程度）
- 数据：尽可能引用具体数字（来源标注）
- 实体：全程使用统一命名（参考 entity.md）
- 目标：每段都能作为 AI 直接引用的独立答案单元

**P0 格式（官网/认证账号风格）：**
```
写 [{BRAND}] 的 P0 级 GEO 内容。

要求：
- 权威、客观、无夸大
- 每个核心产品/功能有独立段落，开头一句直接回答「这是什么」
- 包含可被引用的关键数字（成立时间、规模、荣誉等）
- 实体清晰度优先：同一实体全篇命名一致
- GEO 评分目标：≥ 85 分

目标 Query（来自策略报告）：{QUERIES}
参考 entity.md 中的实体档案。
```

**P1 格式（权威媒体风格）：**
```
写 [{BRAND}] 的 P1 级 GEO 内容，模拟权威媒体报道风格。

要求：
- 有叙事弧：背景 → 现状 → 意义
- 有引语：模拟来自品牌方或行业专家的第三方视角
- 数据支撑：每个重要论点配一个数据佐证
- 适合被 AI 引用的「专家评述」片段

目标 Query：{QUERIES}
```

**P2-3 格式（知乎/公众号风格）：**
```
写 [{BRAND}] 的 P2-3 级 GEO 内容，适合知乎专栏/微信公众号发布。

要求：
- 对话感强，有「Q：... A：...」结构或问答式小标题
- 覆盖用户高频问题（「XX 怎么样」「XX 值得买吗」类）
- 直接答案密度高：每个问题第一段就给结论
- 长度：800-1500 字

目标 Query：{QUERIES}
```

### Step 4：质检

**QC Agent Prompt：**
```
对以下内容按 GEO 评分标准打分，并检查品牌红线。

评分维度（各 25 分）：
1. 直接答案密度：AI 能否从文中直接提取答案？
2. 统计数据量：可引用数字的数量和质量
3. 实体清晰度：命名实体是否清晰、一致、无歧义
4. 结构化程度：标题/列表/表格的合理使用

合格标准：总分 ≥ 70，单项 ≥ 15（P0 实体清晰度 ≥ 22）

品牌红线（用户追加）：{RED_LINES}

输出：
- 每篇得分明细
- 不合格原因（具体到段落）
- 修改建议（可执行，不是泛泛而谈）

不合格的内容打回对应 Writer 返工，最多两次。
```

### Step 5：引用模拟测试

**Simulation Agent Prompt：**
```
模拟 [{BRAND}] 相关内容在 AI 引擎中被引用的概率。

测试方法：
1. 取策略报告中的 P0 级 Query（最高价值问题）
2. 模拟一个 AI 助手在回答这些 Query 时会引用什么内容
3. 对比：我们生成的内容 vs 假设的现有内容，哪个更可能被引用？

输出格式：
- 每个 Query 的预测引用概率（高/中/低）
- 关键引用片段（AI 最可能直接摘录的句子）
- 提升建议（针对引用概率「低」的内容）

注意：这是预测性评估，非真实引用数据。
```

## 多品牌支持

一次跑多个品牌：
```
/geo 歌剧魅影 德国世家 PF79
```

- 各品牌独立处理，互不提及（默认）
- 跨品牌关联需用户手动开启（存在代理协议风险，默认关闭）
- 并行执行，每个品牌一套独立的 Agent 树

## 增量复用

同品牌第二次运行时：
1. 检测 `wiki/{品牌名}/entity.md` 是否存在
2. 加载历史知识库
3. 只更新有变化的部分（差量编译）
4. 新内容追加到 `output/` 对应目录，不覆盖历史输出
